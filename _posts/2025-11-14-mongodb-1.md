```
클러스터(Cluster)
 └── 데이터베이스(Database)
      └── 컬렉션(Collection)
           └── 도큐먼트(Document)
```

```
Cluster (서버)
 ├── Database (DB1)
 │     ├── Collection (users)
 │     ├── Collection (posts)
 │     └── ...
 ├── Database (DB2)
 │     ├── Collection (logs)
 │     └── ...
 └── ...
```

MongoDB 구조 계층
Cluster(클러스터)

= “서버 묶음” 또는 단일 서버(특히 클라우드 Atlas에서는 클러스터 생성=MongoDB 서버/인스턴스 생성)
물리적 또는 논리적 서버들의 집합
(Atlas 콘솔에서 “클러스터 만들기”=DB가 돌아갈 공간/서버 만드는 것)
Database(데이터베이스)

= 여러 개의 논리적 “DB”
하나의 클러스터 안에 여러 개의 데이터베이스(DB)가 존재 가능
(예: myApp, admin, test, …)
Collection(컬렉션)

= RDBMS의 테이블와 거의 같은 개념
각 데이터베이스(DB) 안에 여러 Collection 존재 가능
(예: users, posts, comments, …)
Document(도큐먼트)

= 컬렉션 안에 저장되는 “레코드(행)” (JSON과 매우 유사)
(예: 한 명의 유저, 한 개의 글, …)

### 모델

MongoDB, Mongoos 등에서 모델의 의미
**모델(Model)**은
**어떤 구조와 필드(타입)**의 데이터를 저장/관리할지 “정의”하고,
**실제 저장/불러오기(=CRUD)**와 관련된 함수/메서드를 내장하고 있음

모델 = “User” 정보에 대한 설계도 + 저장소 조작 API 객체

pw123 + salt → 해시
해커가 “해시값”과 “salt”를 모두 알아도
해시 함수는 “단방향”이기 때문에
“역산해서 pw123을 알아내는 공식은 존재하지 않습니다!”
즉,

salt와 해시 둘 다 손에 있어도
→ 원래 비밀번호(평문)는 직접 복구 불가 → 오직 “추측값(후보)”를 해시로 맞춰보는 것(brute-force, 사전공격)만 가능
🟢 결론
해시 + salt를 다 가져가도, 원문(비번)은 알 방법이 없다!
(비번이 아주 단순하거나 쉬운 값이 아니라면 현실적으로 거의 불가능)
